# 日本東京行旅遊系統開發任務書

> 來源文件：`jp_Req.md`, `JP_plan.md`  
> 建立日期：2026-05-28  
> 目標：將旅遊需求規格轉換為可開發、可排程、可驗收的任務清單  
> 建議第一版定位：手機優先的 Web MVP

---

## 1. 專業可行性調整

### 1.1 開發策略

原需求涵蓋 Dashboard、行程、住宿、交通、餐廳、景點、支出、分帳、通知、權限、地圖、天氣、匯出與照片上傳。若一次全部開發，時程與測試風險偏高。

建議第一版採「可查詢 + 可記帳 + 可分帳」作為 MVP，先完成旅途中一定會用到的功能，再逐步加入多人登入、外部 API 與進階匯出。

### 1.2 MVP 保留項目

| 功能 | 是否納入 MVP | 原因 |
|---|---|---|
| Dashboard | 是 | 旅途中最常查看 |
| 每日行程 | 是 | 核心功能 |
| 住宿資訊 | 是 | 6/27 有換住宿，風險高 |
| 交通資訊 | 是 | 航班、Skyliner、富士山交通都需提醒 |
| 餐廳訂位 | 是 | 已有訂位編號與人數 |
| 景點商店 | 是 | 以查詢與地區分類為主 |
| 支出記錄 | 是 | 旅途中即時記帳 |
| 分帳統計 | 是 | 多人旅遊核心價值 |
| 待補資料 | 是 | 需求中仍有未確認資料 |
| 登入權限 | 否，第二階段 | MVP 可先用單一共享系統或本機資料 |
| 推播通知 | 否，延後 | 需額外平台整合，Dashboard 提醒先替代 |
| Google Maps API | 否，延後 | MVP 先使用 Google Maps URL |
| 天氣 API | 否，延後 | 先手動記錄雨天備案 |
| 收據照片 | 否，延後 | 先完成文字支出與分帳 |
| PDF 匯出 | 否，延後 | 先做 CSV 或 Markdown 匯出 |

### 1.3 技術假設

若專案尚未建立，建議採用：

| 層級 | 技術 |
|---|---|
| 前端 | Next.js App Router |
| UI | Tailwind CSS, shadcn/ui, lucide-react |
| 資料 | 第一階段 JSON seed，第二階段 Supabase/PostgreSQL |
| 部署 | Vercel |
| 匯出 | CSV |

若需更快完成，可先不接資料庫，使用本地 JSON seed + 瀏覽器 localStorage 完成 Demo；正式多人協作再切 Supabase。

## 3. UI/UX 設計任務基準

本系統網站設計需參考 `JP_UIUX.md`，採用「Minimal Travel Dashboard Style」，也就是簡潔旅遊儀表板風格。整體方向需兼具旅行手帳感與管理系統效率，第一版以手機使用情境優先。

### 3.1 視覺風格要求

| 項目 | 開發要求 |
|---|---|
| 設計定位 | 簡潔、美觀、易讀、旅行感、管理感 |
| 核心情境 | 旅途中快速查看今日行程、下一站、交通、住宿、訂位、支出 |
| 版面風格 | 卡片式資訊區塊、時間軸、標籤、統計卡、清單 |
| 視覺語彙 | 東京、富士山、車票、地圖、旅行提醒，但避免過度插畫 |
| 裝置策略 | Mobile First，桌機提供較高資訊密度 |
| 深淺色 | 必須支援 Light / Dark / System |

### 3.2 Design Tokens

需建立 CSS Variables 或 Tailwind theme token，避免在元件中散落硬編碼色彩。

| Token | Light Mode | Dark Mode | 用途 |
|---|---|---|---|
| `primary` | `#1E3A5F` | `#7FB3FF` | 標題、主要按鈕、Active 導航 |
| `secondary` | `#4A90E2` | `#8CCBFF` | 連結、交通資訊 |
| `accent` | `#F5A6B8` | `#F2A7C3` | 旅行感點綴、餐飲 |
| `success` | `#5FA777` | `#8AD6A0` | 已訂位、已結清、完成 |
| `warning` | `#F5A623` | `#FFD166` | 待確認、即將出發 |
| `danger` | `#D94F45` | `#FF7A70` | 未結清、取消、高優先 |
| `info` | `#5BC0DE` | `#7FDBFF` | 一般資訊、交通提醒 |
| `bg` | `#F7F8FA` | `#0F172A` | 頁面背景 |
| `surface` | `#FFFFFF` | `#111827` | 卡片、表格、彈窗 |
| `surface-soft` | `#F0F3F6` | `#1F2937` | 次要區塊 |
| `border` | `#E1E5EA` | `#334155` | 邊框與分隔線 |
| `text-primary` | `#1F2933` | `#F8FAFC` | 主要文字 |
| `text-secondary` | `#5F6B7A` | `#CBD5E1` | 次要文字 |
| `text-muted` | `#9AA4B2` | `#94A3B8` | 輔助文字 |

### 3.3 字體與排版

| 項目 | 開發要求 |
|---|---|
| 字體 | `Inter`, `Noto Sans TC`, `Noto Sans JP`, `system-ui`, `sans-serif` |
| H1 | Desktop 32px，Mobile 26px，weight 700 |
| H2 | Desktop 24px，Mobile 22px，weight 700 |
| 卡片標題 | Desktop 20px，Mobile 18px，weight 600 |
| 正文 | Desktop 16px，Mobile 15px |
| 輔助文字 | Desktop 14px，Mobile 13px |
| 標籤文字 | 12px，weight 500 |
| 數字資訊 | Desktop 20px，Mobile 18px，weight 700 |

### 3.4 版面與元件規則

| 元件 | 開發要求 |
|---|---|
| Grid | Mobile 4 欄，Tablet 8 欄，Desktop 12 欄 |
| Spacing | 使用 4px scale：4, 8, 16, 24, 32, 48 |
| Button | 高度 32/40/48px；手機主要 CTA 使用 48px |
| Card | 內距 16-24px，標題層級清楚，重要時間加粗 |
| Tag | 高 24px，圓角 999px，12px 字級 |
| Table | 桌機可用表格，手機需轉為卡片，不壓縮成難讀表格 |
| Form | 必填 `*`，錯誤訊息在欄位下方，手機可用 Bottom Sheet |
| Icon | 使用 lucide-react，手機導航 icon + label，不以純 icon 表達功能 |

### 3.5 導航要求

| 裝置 | 導航 |
|---|---|
| Desktop | 左側 Sidebar，寬度約 240px，可收合至 72px |
| Mobile | Bottom Navigation，高度 64-72px，項目為首頁、行程、交通、支出、更多 |
| 更多頁 | 放住宿、吃與喝、景點商店、設定 |
| Active 狀態 | 使用 Primary 色與淡色背景 |

### 3.6 頁面 UI 要求

| 頁面 | UI 要求 |
|---|---|
| Dashboard | 今日行程、下一個提醒、快速入口、支出摘要需在手機單欄清楚呈現 |
| 行程表 | 日期 Tabs + 時間軸；6/22、6/24、6/27、6/28 需有特殊提醒 |
| 住宿 | 兩段住宿用卡片區隔；6/27 換住宿需醒目 |
| 交通 | 航班與 6/28 回程交通使用大型提醒卡或高優先標籤 |
| 吃與喝 | 已訂位與備選餐廳分區；備選餐廳依地區分類 |
| 景點商店 | 景點、商店、拍照點、神社、模型電器分類 |
| 支出 | 統計卡 + 新增支出 CTA + 支出列表；手機列表用卡片 |
| 分帳 | 每人應付、代墊、應收應付差額需用統計卡呈現 |

### 3.7 互動與狀態

| 狀態 | 開發要求 |
|---|---|
| Loading | 使用 Skeleton Card / Skeleton Rows |
| Empty | 使用簡短友善文案，例如「目前還沒有支出，新增第一筆花費吧」 |
| Error | 明確描述原因，例如「請輸入有效金額」 |
| Hover | 桌機卡片 hover 背景微變、陰影稍加深 |
| Active | 按下按鈕可有 `scale(0.98)` 或輕微色彩變化 |
| Focus | 鍵盤 focus 必須明顯 |

### 3.8 Accessibility 驗收要求

| 項目 | 要求 |
|---|---|
| 對比 | 主要文字與背景需達 WCAG AA |
| 狀態表示 | 不可只靠顏色，必須有文字標籤 |
| 點擊區域 | 手機按鈕至少 44px x 44px |
| Icon Button | 至少 40px x 40px，需有 `aria-label` |
| 表單 | 每個欄位需有 label，錯誤訊息需與欄位關聯 |
| Modal | 開啟後 focus 保持在 Modal 內 |

## 13. 建議開發順序

1. 先完成 Design Tokens、深淺色模式與共用 UI 元件，避免各頁風格分裂。
2. 完成資料型別與 seed data，避免畫面先做完卻沒有一致資料。
3. 再做 Dashboard 與每日行程，快速驗證整體資訊架構與手機版體驗。
4. 接著做住宿與交通，先處理旅遊高風險資訊。
5. 再做支出新增與分帳計算，這是系統真正有別於靜態文件的價值。
6. 最後補餐廳、景點、待補管理、手機版 QA、UI/UX 驗收與部署。

此任務書的重點是把第一版做小、做穩、做得旅途中真的能用。外部 API、登入與推播都可以很漂亮，但不應阻塞核心 MVP。
