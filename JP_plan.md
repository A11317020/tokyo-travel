# 日本東京行旅遊系統開發規劃

> 參考文件：`jp_Req.md`  
> 規劃日期：2026-05-28  
> 旅遊期間：2026/06/22（一）至 2026/06/28（日）  
> 系統定位：多人旅遊行程、交通、住宿、餐廳、景點與分帳管理系統

---

## 1. 規劃摘要

本系統建議先以「旅途中可快速查看與記錄」為核心，優先完成手機友善的 Web MVP。第一版不必急著做完整社群、通知推播或複雜地圖串接，應先確保以下事情能穩定完成：

1. 每日行程可依日期快速查看。
2. 航班、Skyliner、住宿切換與餐廳訂位可清楚提醒。
3. 支出可即時新增，並能計算多人分帳。
4. 未確認資料可被標記，避免旅途中找不到待補事項。
5. 手機版操作簡單，旅途中不用花時間學習介面。

建議採用分階段開發：

| 階段 | 目標 | 交付成果 |
|---|---|---|
| Phase 0 | 需求整理與資料校正 | 確認資料表、待補資料、MVP 範圍 |
| Phase 1 | 靜態 MVP | Dashboard、每日行程、住宿、交通、餐廳、景點清單 |
| Phase 2 | 可編輯 MVP | 新增、修改、刪除資料，支出記錄與分帳計算 |
| Phase 3 | 多人協作 | 登入、角色權限、多人同步 |
| Phase 4 | 進階功能 | Google Maps、天氣、匯率、匯出、收據照片 |

## 2. 原需求修改建議

### 2.1 需求範圍調整

原需求內容完整，但第一版若全部實作會偏大。建議將功能分成「必做」、「第二階段」、「延後」三類。

| 分類 | 功能 | 建議 |
|---|---|---|
| 必做 | Dashboard、每日行程、住宿、交通、餐廳景點、支出、分帳 | MVP 核心，優先完成 |
| 第二階段 | 登入、角色權限、多人同步、資料匯出 | 等基礎資料流穩定後再做 |
| 延後 | 天氣、匯率自動抓取、推播通知、照片與收據上傳 | 旅遊前若時間不足可先不做 |

### 2.2 資料模型修改建議

原文件已列出主要資料表，但建議做以下調整，讓系統更容易擴充。

| 原設計 | 建議修改 | 原因 |
|---|---|---|
| 行程直接以文字記錄交通、餐廳、景點 | 行程表保留 `linked_type` 與 `linked_id` | 可連結住宿、交通、餐廳、景點，避免重複輸入 |
| `participants` 使用文字 | 建議建立 `expense_participants` 關聯表 | 分帳計算更準確，可支援指定分攤 |
| 付款者使用姓名文字 | 改為 `payer_user_id`，畫面再顯示姓名 | 避免同名或錯字 |
| 餐廳、景點、商店分開管理 | 可保留 foods 與 places，但共用地區與地圖欄位格式 | 篩選與地圖開啟邏輯一致 |
| 提醒散落在各模組 | 新增 `reminders` 表或由重要日期動態產生 | 方便集中顯示今日提醒 |
| 匯率只寫在需求中 | 新增 `exchange_rates` 設定表 | 支援 TWD 與 JPY 統計 |

### 2.3 欄位命名修改建議

建議統一使用英文 snake_case 欄位，前端顯示中文標籤。

| 中文顯示 | 欄位名稱 | 備註 |
|---|---|---|
| 優先度 | `priority` | `high`, `medium`, `low` |
| 預約狀態 | `reservation_status` | `none`, `pending`, `confirmed` |
| 票券狀態 | `ticket_status` | `not_bought`, `bought`, `onsite` |
| 結清狀態 | `settlement_status` | `unsettled`, `partial`, `settled` |
| 分攤方式 | `split_type` | `equal`, `custom`, `personal` |

### 2.4 UI 修改建議

原需求建議卡片式設計，方向正確，但旅途中使用情境應以手機掃讀為主。

建議調整如下：

1. Dashboard 首屏只放最重要資訊：今日行程、下一個交通、下一個訂位、今日提醒。
2. 每日行程採時間軸，不要一開始就用大表格呈現。
3. 支出頁可以用「快速新增」按鈕，欄位分兩段：必要欄位與進階欄位。
4. 餐廳與景點用地區分組，避免旅途中滑太久。
5. 未確認事項要有獨立頁或 Dashboard 區塊。

## 3. 建議 MVP 功能

### 3.1 MVP 必做功能

| 模組 | MVP 功能 | 備註 |
|---|---|---|
| Dashboard | 旅遊日期、今日行程、下一個交通、下一個訂位、重要提醒 | 首頁核心 |
| 每日行程 | 日期切換、時間軸、行程卡片、備案 | 以 6/22 至 6/28 為主 |
| 住宿 | 田端住宿、三之輪住宿、入住退房、地址、行李備註 | 需清楚呈現 6/27 換住宿 |
| 交通 | 航班、Skyliner、JR、富士山交通、包車 | 6/28 機場時間提醒為高優先 |
| 餐廳 | 已訂位餐廳、備選餐廳、訂位編號、人數 | Yoroniku 與 THE SUSHI TOKYO 旬必須顯示 |
| 景點商店 | 景點、購物點、地區、優先度 | 支援地區篩選 |
| 支出 | 新增支出、付款者、金額、幣別、分攤人 | 旅途中最常用 |
| 分帳 | 每人總額、代墊、應收應付、未結清 | MVP 需要可用但可先簡化 |
| 待補清單 | 未確認資料、負責人、狀態 | 防止資訊缺口 |

### 3.2 MVP 暫不實作功能

| 功能 | 暫緩原因 |
|---|---|
| 推播通知 | 需要額外平台設定，第一版可先用 Dashboard 提醒 |
| Google Login | 若是小團體使用，可先用簡易代號或單一共享頁 |
| Google Maps API | 第一版可先存 Google Maps URL，點擊開啟 |
| 天氣自動抓取 | 可先人工填寫雨天備案 |
| 收據照片上傳 | 支出文字紀錄先完成，照片第二階段處理 |
| PDF 匯出 | 先支援 Markdown 或 CSV 匯出即可 |

## 4. 建議資訊架構

### 4.1 導覽結構

| 頁面 | 路徑建議 | 主要內容 |
|---|---|---|
| Dashboard | `/` | 今日摘要、提醒、下一個行程 |
| 行程 | `/itinerary` | 每日時間軸 |
| 住宿 | `/accommodations` | 住宿清單與換宿提醒 |
| 交通 | `/transport` | 航班、Skyliner、JR、包車 |
| 餐廳 | `/food` | 已訂位與備選餐廳 |
| 景點商店 | `/places` | 景點、商店、購物點 |
| 支出 | `/expenses` | 支出列表與新增 |
| 分帳 | `/settlement` | 每人應收應付 |
| 待補 | `/todos` | 未確認資料 |

### 4.2 Dashboard 首頁內容

建議首頁由上到下依序呈現：

1. 旅遊名稱與日期。
2. 今日日期與今日區域。
3. 下一個行程卡片。
4. 下一個交通提醒。
5. 下一個餐廳或門票訂位。
6. 今日住宿資訊。
7. 未結清支出摘要。
8. 待補資料提醒。

## 5. 建議資料庫設計

### 5.1 核心資料表

| 資料表 | 用途 |
|---|---|
| `users` | 同行者、付款者、角色 |
| `trips` | 旅遊主檔 |
| `itineraries` | 每日行程 |
| `accommodations` | 住宿 |
| `transportations` | 交通 |
| `foods` | 餐廳與咖啡廳 |
| `places` | 景點、商店、購物點 |
| `expenses` | 支出主檔 |
| `expense_participants` | 支出分攤明細 |
| `todos` | 未確認事項 |
| `reminders` | 重要提醒 |
| `exchange_rates` | 匯率設定 |

### 5.2 建議新增欄位

#### `itineraries`

| 欄位 | 型態 | 說明 |
|---|---|---|
| `trip_id` | UUID | 所屬旅遊 |
| `sort_order` | INT | 同一天內排序 |
| `linked_type` | VARCHAR | 可連結 `food`, `place`, `transport`, `accommodation` |
| `linked_id` | UUID | 被連結資料 ID |
| `is_flexible` | BOOLEAN | 是否為彈性行程 |
| `status` | VARCHAR | `planned`, `confirmed`, `cancelled` |

#### `expenses`

| 欄位 | 型態 | 說明 |
|---|---|---|
| `trip_id` | UUID | 所屬旅遊 |
| `payer_user_id` | UUID | 付款者 |
| `amount_original` | DECIMAL | 原幣金額 |
| `currency` | VARCHAR | TWD 或 JPY |
| `exchange_rate` | DECIMAL | 換算匯率 |
| `amount_twd` | DECIMAL | 台幣換算金額 |
| `receipt_url` | TEXT | 第二階段可用 |

#### `expense_participants`

| 欄位 | 型態 | 說明 |
|---|---|---|
| `expense_id` | UUID | 支出 ID |
| `user_id` | UUID | 分攤人 |
| `share_amount` | DECIMAL | 應分攤金額 |
| `is_settled` | BOOLEAN | 是否已結清 |

#### `todos`

| 欄位 | 型態 | 說明 |
|---|---|---|
| `todo_id` | UUID | 待辦 ID |
| `trip_id` | UUID | 所屬旅遊 |
| `title` | VARCHAR | 待補項目 |
| `category` | VARCHAR | 住宿、交通、餐廳、支出等 |
| `priority` | VARCHAR | 高、中、低 |
| `owner_user_id` | UUID | 負責人 |
| `status` | VARCHAR | `open`, `doing`, `done` |
| `due_date` | DATE | 期限 |

## 8. 技術方案建議

### 8.1 快速 MVP 方案

| 層級 | 建議 |
|---|---|
| 前端 | Next.js 或 React |
| UI | Tailwind CSS、shadcn/ui |
| 資料 | 先用 JSON seed 或 Supabase |
| 部署 | Vercel |
| 驗證 | 第一版可先不登入，使用共享網址 |

適合情境：希望旅遊前快速做出可用系統。

### 8.2 正式 Web 系統方案

| 層級 | 建議 |
|---|---|
| 前端 | Next.js App Router |
| UI | Tailwind CSS、shadcn/ui、lucide-react |
| 後端 | Next.js API Routes 或 Server Actions |
| 資料庫 | PostgreSQL / Supabase |
| Auth | Supabase Auth 或 Google Login |
| 檔案 | Supabase Storage |
| 地圖 | Google Maps URL 起步，後續再接 API |
| 匯出 | CSV 第一優先，PDF 第二優先 |

建議第一版即使選正式技術，也先避免過度工程化。資料表設計要正確，功能可以分批完成。

## 9. 開發里程碑

### Phase 0：資料與需求整理

預估：0.5 至 1 天

交付：

1. 確認 MVP 功能範圍。
2. 建立旅遊基礎資料 seed。
3. 補上同行者代號。
4. 整理待補清單。

完成條件：

| 驗收項目 | 條件 |
|---|---|
| 日期完整 | 顯示 2026/06/22 至 2026/06/28 |
| 資料可匯入 | 行程、住宿、交通、餐廳、景點、支出可用同一格式管理 |
| 待補可追蹤 | 未確認資料都有狀態 |

### Phase 1：靜態 MVP 介面

預估：2 至 3 天

交付：

1. Dashboard。
2. 每日行程時間軸。
3. 住宿、交通、餐廳、景點頁。
4. 手機版排版。

完成條件：

| 驗收項目 | 條件 |
|---|---|
| 手機可讀 | 主要資訊不需橫向捲動 |
| 今日資訊清楚 | Dashboard 可看到今日行程與提醒 |
| 關鍵資料完整 | 航班、Skyliner、住宿、訂位餐廳都可查 |

### Phase 2：資料編輯與支出分帳

預估：3 至 5 天

交付：

1. 新增、修改、刪除行程資料。
2. 新增支出。
3. 分帳計算。
4. 未結清狀態管理。
5. CSV 匯出。

完成條件：

| 驗收項目 | 條件 |
|---|---|
| 支出可新增 | 可輸入金額、付款者、分攤人、幣別 |
| 分帳可計算 | 顯示每人應收與應付 |
| 結清可追蹤 | 可標記已結清或未結清 |

### Phase 3：多人協作與權限

預估：3 至 5 天

交付：

1. 登入。
2. 使用者角色。
3. 權限控管。
4. 多人同步資料。

完成條件：

| 驗收項目 | 條件 |
|---|---|
| 使用者可登入 | 可辨識管理者與同行者 |
| 權限正確 | 一般查看者不可修改 |
| 支出歸屬正確 | 本人新增項目可追蹤 |

### Phase 4：進階功能

預估：視需求追加

候選功能：

1. 天氣與雨天備案。
2. 匯率自動更新。
3. Google Maps API。
4. 收據照片上傳。
5. PDF 匯出。
6. 提醒通知。

## 11. 風險與因應

| 風險 | 影響 | 因應 |
|---|---|---|
| 需求一次做太多 | 開發時程拉長 | 先完成 MVP，進階功能延後 |
| 分帳規則未確認 | 計算結果不準 | 先確認同行者、共同支出與個人支出規則 |
| 旅遊前資料未補齊 | 提醒不完整 | 建立待補清單並在 Dashboard 顯示 |
| 手機版資訊過多 | 旅途中不好用 | 首頁只顯示下一步與重要提醒 |
| 外部 API 設定耗時 | 阻礙 MVP | 地圖、天氣、匯率先用手動資料 |

1. 先確認是否要開發成 Web App，或先做 Google Sheets / Notion 版本。
2. 補齊同行 5 人代號，這會影響分帳設計。
3. 補齊三之輪住宿與富士山包車資料。
4. 決定 MVP 是否需要登入。
5. 若要開始開發，建議先建立資料 seed，再做 Dashboard 與每日行程頁。

---

## 14. 結論

此旅遊系統最重要的價值不是功能多，而是旅途中能快速回答三個問題：

1. 現在要去哪裡？
2. 下一個不能錯過的時間是什麼？
3. 誰付了錢，最後要怎麼分？

因此第一版應聚焦在 Dashboard、每日行程、重要提醒與分帳。等這些核心流程穩定後，再加入登入、多人同步、地圖、天氣與匯出等進階功能，會是比較穩妥也最實用的開發路線。
